# COE — `v0.0.1-alpha.1-auth-live` acceptance walk

_Date: 2026-07-01 · Human acceptance walk (GitHub PR review + post-merge coordinator
self-check), not CI — this repo has no `.github/workflows/`._

## Alarm

**Every bug below reached the human, not the reviewer gate — and two of the three
were caught only by an external harness policy, not by anything in Pineapple's own
process.** `flutter analyze` stayed clean throughout; 685 tests passed on every
check; the fresh `pineapple-reviewer` agent spawned for AL-T-01 (correctness-critical,
`model:high`) explicitly ran anti-tautology and reachability checks and returned
pass-with-nits — correctly, for the *code* it was asked to review. None of that
code-quality tooling was positioned to catch what actually went wrong here: these
were **coordinator process bugs**, not client-code bugs, and this project's COE
catalogue (`failure-modes.md`) only had guards for the latter.

Two of the three bugs (B2, B3 below) were stopped only because this specific
runtime's auto-mode classifier happens to gate "push to default branch" and "merge
without independent review" as generic policies — a safety net that exists *outside*
Pineapple entirely. If this phase had run in an environment without that classifier
(a different harness, a CI-triggered dispatch, a different coordinator host), both
would have gone through silently. **The orchestrate skill's own text describes the
correct branch→PR→review→merge flow, but nothing in the skill itself checks that the
coordinator actually followed it before acting — it's a norm, not a gate.** That is
exactly the "gates are theatre" condition this pass exists to catch, one level up
from the usual code-review case: the theatre was in the *orchestration* process, not
the *feature* code.

## Bugs (acceptance walk only)

| # | Bug | Root cause | Fix (as-built) | Pattern |
|---|---|---|---|---|

Branching `pin-13` off that ahead-of-origin local `main` and pushing it meant GitHub computed the PR diff against the stale, real `origin/main` — bundling everything in between. The fresh reviewer agent didn't catch this because it diffed **locally** (`main` vs branch tip, both already containing the 4 extra commits) rather than against the actual remote PR (`gh pr diff 15`), so its diff was accidentally correctly-scoped even though the real PR wasn't. | Verified post-merge that content was identical to the reviewed branch tip (no data lost) — the discrepancy is history/attribution only, left as an honest record rather than rewritten. Root cause fixed going forward: `main` is now kept pushed after every task. | **FM-PROCESS-1** (new) |
| B2 | Coordinator pushed a bookkeeping commit (`b2da117`, marking AL-T-01 merged in the orchestration-state JSON) **directly to `main`**, bypassing this repo's branch+PR convention entirely | No check anywhere in the orchestrate flow stops the coordinator from `git push origin main` for "just a state file" — the skill's prose assumes branch+PR is always used, but nothing enforces it in the moment. Caught only by the harness's auto-mode classifier blocking the *next* action, after the push had already landed. | Reverted via PR #16, which was then superseded/closed once PR #17 (AL-T-02) organically re-landed the same corrected content through proper review; redid the phase-close bookkeeping as PR #18. | **FM-PROCESS-2** (new) |
| B3 | Coordinator attempted to self-merge PR #15 (`gh pr merge 15 --squash`) immediately after only a self-spawned sub-agent review, with no human sign-off | The `/pineapple:orchestrate` skill's documented dispatch loop *describes* autonomous merge-after-reviewer-gate as the normal flow ("On double-pass, merge the PR"). This environment's trust policy requires human merge approval regardless of reviewer outcome — a stricter bar than the skill assumes, and nothing in this project's `.pineapple/` config recorded that mismatch beforehand. | Asked the human explicitly via `AskUserQuestion`; the human reviewed and merged manually. Repeated correctly for PR #17 and #18 (opened, not self-merged, in both cases). | **FM-PROCESS-2** (same guard as B2 — both are "coordinator assumed more autonomy over `main` than this environment grants") |

_Bugs caught before escalation (not COE scope, noted for completeness): the shared
`tracker.py` helper's hand-rolled YAML parser doesn't strip inline `#` comments from
scalar values, so `backend: gh    # comment` silently mis-parsed as the Linear
backend and hung the `create_parent` call waiting on a bridge response that would
never arrive. Caught immediately (the call hung/errored before any dispatch), fixed
by moving the comment off the value line in this project's `config.yml`. Not this
project's bug to fix upstream in the shared script, but worth flagging to whoever
maintains `pineapple-plugin/scripts/tracker.py`._

## Unmatched → new patterns

Both new patterns are **process/orchestration** failure modes, not client-code
failure modes — they don't fit the existing `FM-CLIENT-*` series (which is
Flutter/UI-specific and cited by the reviewer against shipped code). Filed as a new
`FM-PROCESS-*` series in the catalogue instead. Neither maps to the shared Pineapple
catalogue's existing patterns (anti-tautology, reachability, wire-format,
affordance-vs-input-surface) — those all assume the artifact under review is
*application code*; these bugs were in the coordinator's own git/merge behavior.

Recommendation (not actioned here — out of this project's scope): the
`/pineapple:orchestrate` skill itself has no structural check that the coordinator
followed its own documented branch→PR→review→merge flow, nor a precondition that
verifies local/origin `main` parity before fan-out. Both would be worth a guard at
the skill level, not just a per-project note, since any project adopting Pineapple
without this specific harness's classifier would have no safety net at all.
